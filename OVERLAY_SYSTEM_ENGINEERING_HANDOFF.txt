Project: Project-Phoenix (React 19 + Vite)

Theme: Centralized overlay hosts (#dropdown-root, #popover-root, #tooltip-root,
#modal-root, etc.), Z_INDEX in src/constants/zIndex.ts, mountOverlayHostStack() /
getOverlayHost() in src/utils/overlayHosts.ts.

This report summarizes overlay-related work (completion pass + residue sweep) so
reviewers can trace what changed, why, and what stayed local on purpose.

--------------------------------------------------------------------------------
1. GOALS
--------------------------------------------------------------------------------
- Stop clipping / wrong stacking for menus, popovers, tooltips, and comboboxes
  inside cards, masonry, scroll areas, and overflow-* / transform ancestors.
- No ad-hoc z-[9999] or "raise z-index until it works."
- Prefer portaling to named hosts + existing primitives: DropdownPortal,
  ModalShell, getOverlayHost, shared Tooltip where appropriate.
- Preserve a11y (roles, keyboard, focus) and outside-click / Escape behavior.

--------------------------------------------------------------------------------
2. ARCHITECTURE (unchanged intent)
--------------------------------------------------------------------------------
- index.html: sibling roots for overlays (dropdown-root, popover-root,
  tooltip-root, drawer-root, modal-root, ...).
- mountOverlayHostStack() (app bootstrap): applies position: fixed; inset: 0;
  pointer-events: none and z-index from Z_INDEX.
- Floating content sets pointer-events: auto where it handles input.

Host choice (documented in overlayHosts.ts):

  dropdown  - Action menus, column pickers, card "more" menus
  popover   - Richer anchored panels (+N tags, bookmark menu)
  tooltip   - Hints + interactive bubbles that must escape clipping (e.g. author popover)
  modal     - Dialogs via ModalShell; combobox list inside an open modal uses
              DropdownPortal host="modal" so the list stacks above the modal body

--------------------------------------------------------------------------------
3. DropdownPortal API enhancements
--------------------------------------------------------------------------------
File: src/components/UI/DropdownPortal.tsx

- placement: 'below' | 'above' (e.g. masonry HUD "more" above anchor).
- host: 'dropdown' | 'popover' | 'modal' (typed as AnchoredOverlayHost).
- matchAnchorWidth: combobox-style width alignment.
- overlayRootProps: optional HTML attrs on the portaled root (e.g. listbox id,
  role, aria-label), with onClick chained so stopPropagation is preserved.

Docstring points engineers to overlayHosts.ts for layering rules.

6. INTENTIONALLY NOT PORTALED (by design)
--------------------------------------------------------------------------------
- Full-screen / modal-internal UI: ImageLightbox, controls inside YouTubeModal /
  ModalShell children -- absolute + local z-* is acceptable within that surface.
- Card/media chrome: selection chips, HUD on thumbnails, gradients -- not global
  menus; in-layout is OK if they do not escape the card.
- Drawers / mobile sheets / admin mobile sidebar -- separate from dropdown hosts.
- src/_archive/** -- legacy experiments; not part of production overlay migration.

--------------------------------------------------------------------------------
7. menuRef + useNewsCard mousedown (still there)
--------------------------------------------------------------------------------
CardActions keeps a wrapper ref (menuRef) because:
- MenuDropdown is portaled but tagged data-card-actions-menu="true".
- useNewsCard outside-click logic ignores clicks inside that panel while still
  treating the trigger area as part of the card menu flow.

This is architectural, not a z-index hack.
